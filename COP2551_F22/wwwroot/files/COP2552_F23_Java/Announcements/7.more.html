Let me know if my hints and discussions on Discord for Project 3 were helpful.

 

I went through all of the Project 2 submissions.  I think everyone realizes by now that the more time (we) spent writing code, the easier this gets. And that the to complete the project assignment satisfactorily, it's more than just a single sitting.  The process of elaboration is definitely helpful. Get something (minimal) working, and add to it. The "big-bang, write-it-all and fix the errors" is less likely to work.

One observation I've heard, and I agree with, is that the instructions (requirements) are not always as clear as we would like, but that is an important part of the task too. More often than not, we get incomplete or even misleading requirements from a customer and working through those requirements is sometimes an imperfect science.

 

A couple of reminders from the projects I saw in Project 2, ALWAYS avoid using unnamed constants. If the input patient data file for week 3 is "PatientDataW3.txt" consider naming all of your constants together in one place near the top of your code:
outputDataFilePath = "C:\\SFC\\COP2552\\Project2";
inputDataFile = getInputDataFileName();   // look for newest patient data file and get confirmation from user
outputDataFile = getOutputDataFileName(inputDataFileName);  // increment week # based on current input data
patientDataErrorFile = getErrorDataFileName(outputData);

This level of detail wasn't required, but it means that we don't have to rewrite the application every single week when the data file names change.

 

Read the instructions and then read them again.  If the instructions indicate "using good object-oriented programming principles that will demonstrate the use of multiple classes," identify what those classes are. As you write similar classes, are there common properties or methods that could be written to a base class and inherited?  If you are repeating code, then you probably missed something.

 

Writing code every day doesn't mean staring at the same project for hours at a time. Code (not copy) the samples from the chapter. Change them up. What are they trying to demonstrate?  What can you apply to your project?  Every time you work on your project, try and end the session with a working application, even if it doesn't meet all of the requirements yet. Keep adding to it until it's something that is maintainable, self-documenting, logical and stable.
