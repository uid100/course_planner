Formatting

Every language has syntax constraints which are largely managed by the compiler or interpreter. Yes, logic errors may still get in the way of "correct" results, but if the code is not syntactically correct, execution will fail.

In addition to syntax, there are code conventions. While it is true that coding conventions may vary some between engineering teams, it is becoming more common for style-guides to follow language-specific guidance, with minimal to no local tailoring.

 

ok, but what does that mean?

More simply put, my company may have a very specific template for source code file headers to include "properly" formatted copyright information, documented baseline pedigree, author information, and file change history, and such, but more and more commonly, C#, or Python, or Java, or javascript files will adhere very closely with industry standards.

For instance, Python code usually uses snake_case for identifiers (using underscores as word separators), where C# and Java developers favor camelCase for variables and PascalCase for class names. Most IDEs are configured to format spacing according to these language-specific guidelines. 

It's important to know that these conventions, or guidelines are rules in name only. A block of code may be functionally correct but not be consistent with these community standards. So while the code may work, it may still be considered incorrect by the project team.

 

But if it works, why does it matter?

On a project team, in a production environment, every document, code segment, or other artifact will be peer-reviewed before being released either internally or to customers. If this isn't universally true, then any variance should be seen as an (incorrect) exception. As I discussed previously, these peer-reviews are the team's responsibility. So, imagine a group of three engineers, billing their time at a very expensive rate, spend several hours reviewing each other's code, and formally documenting findings that amount to things such as whitespace and comment formats. Next the author reviews the changes and decides he or she preferred the way it was before, and restores the formatting to his or her own styles. How many needless hours are spent making changes to code incurring great cost to the company and customer with no value added to the product. That's the first, but not the only problem.

Next imagine a defect (or three) is (are) found after the code has been released to production, or a new feature is contracted and changes need to be made to the code baseline. If the code is written to the accepted guidelines, then a developer that steps in to fix it will have a much easier time coming up to speed with the original code and making the required changes. Sometimes, it could be you that are asked to fix code that was written weeks or even months earlier. Well-written code is more readable, and more maintainable, and is less prone to errors.

Whitespace

"... is any string of  text composed only of spaces, tabs or line breaks (to be precise, CRLF sequences, carriage returns or line feeds). "

Judicious use of blanks and blank lines will improve readability. Blank lines between methods are helpful. Several consecutive blank lines sprinkled randomly through a file are not. Check the code convention for where they belong and be consistent in your code.

 

Line Length

Most common coding conventions recommend limiting lines of code to less than 80 columns. Some guidelines recommend 70, although this can be a challenge with nested indented blocks. Modern monitors will support wider displays than that, but when your code is spanning beyond 100 columns, it's probably going to wrap in most editors and readability suffers. This is also much harder to maintain. I saw some examples in your projects where lines spanned over 220 columns!

/*-------1---------2---------3---------4---------5---------6---------7--------*/

Most editors will display the row and column number of the cursor, but this line used to be commonly placed throughout code projects.  I still use it from time to time while I'm editing big projects, although I will usually delete all of them before committing the final changes to production.

<!-------1---------2---------3---------4---------5---------6---------7--------->

This is an .html version.

These are not strict rules. Sometimes, breaking the line at 80 columns is a very awkward breakpoint and maybe lines need to go to 86 or 91 to be clear. But when they start getting too long to reasonably be viewed on a laptop for instance, keep those as an exception, and use the guidelines as objectives.

 

Comments

Comment formatting is also covered in the code conventions, but when it comes to comments, how much is enough? Is there such a thing as too much?

Consider the following:

Triangle.java

public class Triangle {
    private double base;
    private double height;

    public Triangle(double b, double h) {
        base = b;
        height = h;
    }

    public double Area() {
        return (this.base * this.height) / 2.0;
    }
}

Can you tell what this code does, what the methods are for and what the variables contain?

 

Here's another:

Distance.java

public class Distance {
    float MI_PER_KM = 0.62f; // Conversion factor from customer requirements

    public double milesToKm( double miles ) {
        return miles / MI_PER_KM;
    }

    public double kmToMiles( double kilometers ) {
        return kilometers * MI_PER_KM;
    }
}

In the first example, there are no comments yet the intent and usage are clear. In the second example, there is one comment, indicating the source for a named constant. If the source was not indicated, perhaps someone might decide that 0.621371 miles-per-kilometer is better for precision, but since the constant was specified as a customer requirement, any well-meaning change would change the results and introduce new errors and unexpected results. This comment not only adds value, but leaving it out opens the code to potential errors.

 

Self-documenting code

When identifiers (variable, method, class names) are clear and descriptive, code is more readable, maintainable and less error prone.

Processing should be logical but not complex. Just because you can mathematically stack many calculations onto a single line of code, doesn't mean you should. Does it improve readability?

Comments are very helpful in many situations, but you will find that making them too verbose can also get in the way of understanding and can obscure the code. Sometimes less really is more. If your comment is restating what is already clear in the code, it is probably not adding value. Remember that if the code changes, the comments will likely need to be modified also.

 

Think about the next person to open your code. If someone has to reuse or modify your code two-, three-, six-, or eighteen months from now, will the code's meaning be clear. Too often, we are the ones that have to fix code that we wrote in the past. If you come back to something you wrote in the past and don't remember what it did, or can't determine from the context, does it have the right amount of comments? Also, if too many comments are needed, consider refactoring your code or renaming your identifiers to improve the level of self-documenting code.

Good code is good code, and the more you practice this, the more code you see, the more you study, the better your code will be. Getting it working isn't the same as done.
