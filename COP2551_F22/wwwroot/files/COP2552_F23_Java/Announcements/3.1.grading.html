Code Review

An essential part of any engineering effort, is peer review. This activity occurs continuously throughout the project life cycle, in addition to very specific intervals. It affects all engineering artifacts including requirements and design documents, code, and even test procedures. Not only is peer review its own specialty with its own documented process, it is deeply ingrained in all phases of business and engineering. In most professional environments, nothing will pass from one phases to another, or be released internally or externally, without formal or informal peer review.

 

What Code?

As a new developer on the engineering team, you are most likely to be given a block of working, legacy code and a problem report. This assignment may take on different names from organization to organization, but they are roughly the same. There is an error, or a discrepancy, or performance issue, or new feature request in the system that we are already using, and your first task is to locate it. Notice that the new guy is usually not asked to meet with a customer and design a new solution from the ground up. We're going to bring you up to speed first. Back at your desk, you familiarize yourself with how the system works, try out all of the inputs and try to simulate the same conditions as the problem report and see if the error occurs as described. You may be asked to further refine the description.

 

Change Process

Once you've duplicated the error, you may be given your own separate copy (branch) of the source code to read and follow and step through to identify why the application is responding the way it does, where the values are occurring and make a suggestion about how to change the code to prevent or change the behavior. This results in a change request. Next, the change request will be validated by a senior engineer, resources allocated, and someone, maybe you, will be approved to implement the change. After the code is changed, it will be code reviewed and integrated with the rest of the system, still on that separate branch, and tested thoroughly. Once the change has been tested, then it will be merged with the production branch, tested once more and released.

This is an even more detailed process than what I've described, but the important message is that no code should ever be written or accepted without code review.

 

Code Review

If all of the code that you write, as well as others is reviewed first, who is doing that review? The answer is "everyone."

Code reviews are not personal. As often as a programmer's code is being reviewed, he or she are just as likely being asked to review other's code.

The code needs to be up to the quality standards of the company's products. The reputation of the engineers, the products, and the company depends on it.

    • Engineers are not being reviewed, their product is.
    • Engineers are not being reviewed, their product is.
    • Engineers are not being reviewed, their product is.

If you're the new guy with no experience, or the seasoned professional with many years' experience, that code must be correct, of good quality, and maintainable in order to be acceptable. As an engineer in the organization, you must not care who wrote the code, only that it is the best your company can produce. 

So if everything I write is reviewed, does that mean I should have thick skin, and be prepared to get beat up by regular evaluation, scrutiny, and ridicule? Of course not. Peer review is a good thing. It's an opportunity to learn from others, to help each other develop a better product.

Sometimes new developers are surprised to learn that they are asked to review more experienced developers' code. You may learn as much from looking at code as having others look at yours. 

But what are you (we) looking for?

The code is checked to see that:

    • the requirements are met,
    • the syntax is correct,
    • the logic is correct,
    • the identifiers (variable, constant, and method names) are descriptive and meaningful and consistent with coding standards,
    • that use of whitespace is reasonable for readability and consistent.
    • There are no unnamed or unexplained constants (numeric or string literals)

 

This list is approximate off the top of my head. In most professional environments, the code reviewer would have a formal checklist and the formal coding standards, but this is what you will look for.

 

Or put even more simply:

    • Does it work?
    • Does it do what it’s supposed to do?
    • Is it easy to understand? (and)
    • How could we make it better?

 

In that professional environment, the result of the peer review (or code review) is a list of findings. Each finding will then be reviewed (sometimes by committee) and either:

  • implemented (that's correct, let's fix that)
  • deferred (that's a good idea, but it's not part of the requirement for this assignment and we don't have time or budget right now)
  • or rejected (it might be incorrect because the reviewer didn't have all of the information, or may be inconsistent with another requirement)

The findings are each closed and the cycle continues.

As part of a college computer class, the natural assumption is that the student is being assessed, but really it's the same process. The instructor is looking at the submitted assignment, not the student. It is my goal to learn with you and help you to become better engineers. Prior to this class, you already have learned programming. You've already successfully written applications in other programming languages. I would like you to be better programmers from the experience of programming. I can help by looking at your code and providing feedback. So, 

    • Does it work?
    • Does it do what it’s supposed to do?
    • Is it easy to understand? (and)
    • How could we make it better?

 

A common task in technical interviews is to walk through a so-called white-boarding exercise. The candidate is given a problem or set of requirements and then drafts a solution with either code or pseudo-code, checks the solution, and then (importantly) optimizes the solution. How could you make it better?
