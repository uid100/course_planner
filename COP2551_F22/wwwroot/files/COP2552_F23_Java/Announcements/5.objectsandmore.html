Week 5

I have so many questions!

As we dive into week 5, and each new module for that matter, take a moment and start with the table of contents. Go back and look at the headings for each of the chapters up to this point (Chapters 1 through 5.) You've seen all of these topics before in courses leading up to this. Are any of these different in Java? Do some of the concepts make more sense now from a different perspective?  Do you still have the textbooks you used when you learned other programming languages, C++, C#, javascript, Python, others?  While you're considering this topical outline that is the table of contents, compare it with those other books. What do you think of this author's "early objects" approach? Does it make more sense now? Do you agree with this approach? As an advanced programming student, do you see the logic behind this learning path? Do you remember one topic that you learned that was a "lightbulb" moment, something that suddenly made something else make sense?

As a professional developer and software engineer, one of the lessons that may become clear to you, is that what you are learning, is not just a common language, but how to communicate and stay relevant with an evolving technology landscape. School can prepare you to start a career, but that transition is only a beginning, not an end to what you will learn. The languages, the patterns, the tools, the frameworks that we use, are all under continuous change.

Think about the different software packages that you use every day. How many are there? Eclipse, Windows, MS-Office, Even Instagram, Google Chrome, Google Search, and Netflix.... All of these and many more are always being worked on, updated, fixed, improved. Some of the updates are visible, some of the changes can be easily seen, interface changes for instance, bug fixes. Others may be hidden from view, but still they are being modified, tested, built, and pushed to production. When you start to realize how much software is behind everything around you and how quickly it is all changing, you begin to understand and appreciate that what you are learning when you study programming languages, is not how to make a computer say "hello world," but rather the basic building blocks for building something bigger and how that changing landscape will continue to grow and change.

Writing Code

Textbooks which accompany technical topics, like programming languages, are as much references as they are end-to-end instructions. You may start from the beginning and work your way through them, but you are as likely to use them as a reference to a particular topic. The objective is to learn a skill as much as a concept. It's good to understand how and why something works, but we also need to know how to apply that technology. 

The second point I want to impress on you is to practice what is in textbook. These books are meant to be opened alongside your computer. Try these things on your own. It's not enough for a football team to watch the films (tapes) of last week's game. It's good to recognize ideas and problems, but just like any foreign language, the best way to effectively learn it is to practice using it. Run all of the examples in the book. Mix things up a bit. What happens if I change this or that? Can I run this on another computer? Does it behave differently from the command line than the IDE? What else does this integrated development environment do? How does it add to my code? Are there things that the IDE is doing to make my life easier, make my code better? 

Number two: practice! (A lot!)

It is my opinion that there are not enough graded coding exercises in this course. That doesn't mean you shouldn't be writing more. 

How fast can you create an application, even "hello world" in Java? .. using the IDE? .. using a text editor and command line tools? .. what about using your other favorite language(s)? If you can't do this quickly and confidently, then practice until you can.

Origins and Evolution

Textbook and even online examples sometimes mention component package versions. Do you stop to ask why? If the language and the technology is evolving, maybe some pieces didn't exist in early editions. Maybe somethings won't be supported in later versions. Sometimes it's good to question not just when something in the language changed, but why did it change.

Many if not most textbooks that I've used start with a historical background of how a language came to be. I have often suggested that students skip over this material on first reading. But often after spending some time getting to know a new language or technology, it's a good idea to go back and revisit that. When you were first learning to write code, you probably didn't care, but maybe after a while, it might help make things clearer. Why was Java created? What else was going on at the time? Java came with it's own runtime environment. Why? Is that important?
