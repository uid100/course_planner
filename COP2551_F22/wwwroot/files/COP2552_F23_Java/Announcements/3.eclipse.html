Decisions

As you have seen over the past couple of weeks, although this is a new language for most of you, this course is not a beginning programming class but rather an introduction to a new syntax for familiar concepts. In the first few chapters, the author discussed some history and environment for the Java language, and an overview or review of classes and objects. This week we'll review the concept of conditional logic. Don't get lost in the details. Decision logic is something that should be very familiar. 

IF I am cold THEN I will wear a jacket ELSE I'll probably be comfortable in shorts and a t-shirt.

IF I have an extra thousand dollars after I pay my rent and utilities, THEN I will buy steak and wine for my roommates.

IF the user selects from the menu: (P)lay, (S)top, (R)repeat, (N)ext, THEN decide what to do next based on the input....

 

These are all conditions. In all of our experiences, we consider the inputs and decide what to do next. If you consider all of these as simple Yes or No  decisions, that is the basis of conditional logic. At it's simples form, all decisions are really binary. If you are in a restaurant and presented with a menu of 100 items with options, each item is either yes or no. Even when you decide that 14 of them could be Yes, you finally narrow it down to one as yes. And after considering other decisions, you could decide that more than one could be a Yes also, but ultimately, everything gets decided either yes or no, true or false. This is binary logic, and whether you recognized it or not, you've probably done it every day.

 

The Unified Modeling Language (UML) standard describes 14 different engineering diagrams. You are probably familiar with several. Half of these are structural. These are comprised of static models describing the elements of a system, the class model for instance. The rest are behavioral and describe what happens in the system like use-case diagram, sequence, activity, and state machine diagrams. Most engineering teams will use a subset of these for their system design and planning activities. It has never been my experience to see more than a few applied to any given project. The Sequence diagram, which is (a more standardized form of the familiar flowchart) defines the conditional block with a single entry point, a condition statement, and two (binary) alternative exit directions which are dependent on the evaluation of a condition.

 

This week we'll look (AGAIN) at conditional statements. While we do, I will ask you to spend an hour, considering all of the "conditional" decisions that you make. Do I want pepperoni pizza for dinner? Will an hour at the gym make me feel better? If I called and talked to my mom on the phone for thirty minutes, would it make her happy?

 

IF-ELSEIF-ELSE and SWITCH-CASE logic can be that simple. Take away the syntax for moment and consider the True-False logic statements as nothing more than a simple path with two possible outcomes. Consider that all decisions, and all digital logic paths, regardless of how complex, can be decomposed to that simple, binary logic Think in system-logic.

 

Yes, this week we will be focused (in sync with the textbook) on simple conditional statements and the required syntax to "make it work" in the Java programming language. But it's far more important as we learn the concepts, to generalize these and consider how would you accomplish these in another familiar language. What does the logic look like in English? in C++? in Python? in Java?  How are they different? Or are they really?

 

To be successful, yes it is important to understand how to apply the concepts to a specific environment and language, but more important to consider how the logic applies and why the logic is configured in a particular way.

As we go through this material, you may discover that I inherited much of this curriculum material from another. I may have edited and modified things, but a lot of it is not my original work. This includes the selection of the Eclipse IDE. There are a handful of IDEs that I like to use, depending on the project and language I’m using. Eclipse, Bluej, NetBeans all are good, but I do like to test cross-platform and usually rely on command-line environments for this. In fact I like to develop Java code using Visual Studio Code. This is not meant to replace or override the course material, but rather, to expand it. 

When a requirement, whether explicit or implied, indicates that something is "cross-platform" compliant or inclusive, it suggests (or insists) that the solution is tested on multiple platforms. When I am developing, I usually test on multiple platforms. If I am developing a web solution, I will test on multiple common browsers, not just my favorite. If I am writing code in Python or Java, I will test the solution on different operating systems.

This week a question from a student asked, If I am only submitting a single .java source code file, how can I test this outside of the Eclipse environment? (Great question!)

As, I mentioned, testing in multiple environments is a good idea and taking it to its simplest form is usually ideal.

 

Go to your development folder and run:

javac Markup.java

 

This will (should) create the Markup.class solution file.  Next run:

java Markup

 

notice you are calling the class "Markup," not the class "Markup.class"

This should launch your program solution.  when in doubt, start with the command line.
