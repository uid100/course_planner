Language

Aloha mai kākou!

In the Hawaiian language ( 'õlelo ) and traditions, aloha is much more than a catchy phrase on a t-shirt or bumper sticker. It originates from the word 'alo' which is "to face" or "be present" and "ha" which is a breath. Together, "aloha" is the presence of breath, the first signs of life we exhibit after birth, and the shared experience of life itself. "Aloha mai kākou" is a collective greeting, it is a shared aloha between all of us.

I enjoy traveling when I can and the Hawaiian islands are one of my favorite destinations. I have spent a lot of time on most of the major islands. Generally, I think that a study of language offers better insight to a people and a culture.

This is a language class, do you think the same concept can be applied to computer programming? The Java programming language has a history and an evolution. The language development is steered by people and experiences and influenced by other languages and even cultures and environments. It's an interesting thought. Consider the programming languages that you've used and the (usually simple) examples that you've coded. Can you see the patterns and parallels between them? Do you ever stop and ask why there are differences and what drove the development?

Java and Project 1

JOptionsPane

One student had an interesting question about the use of JOptionsPane in Chapter 2 and I thought I'd share my response with the rest of you.

 

What is the difference in using JOptionsPane and JFrame? I have programmed GUIs in Java prior to this class and always used JFrames.

The short answer is that JOptionsPane is a simple dialog box. It's a way to get information to the user very simply and very quickly. One of the first things to notice is that it's a modal dialog box. That means it's a blocking action. The dialog will stop program execution until it is acted on and dismissed. The two classes also have different inheritance. If you refer to the Oracle docs, you can see that they have different intended uses and purposes.

It's a great question, and we will use both, but for these first exercises, the "keep it simple" mantra prevails. Follow the examples from the book.

Conventions

The instructions for Project 1 include:

  • >Follow all good programming naming conventions.

What are naming conventions?

There are case styles.

    • camelCase
    • PascalCase
    • snake_case
    • kebab-case

Languages usually have their own standards. Snake_case (separating with underscores) is commonly used in Python code, kebab-case, hyphenated is often seen in javascript and CSS.

camelCase and PascalCase are usually included as part of the language coding standards for C# and Java.

Self-documenting code

That's a good start, but there's more.

Self-documenting code also means using descriptive variable names. Consider the following: 

int numberOfStudents;
float heightInInches;
static final float CM_PER_INCH = 2.54;

None of these lines of code contain any comments, but it is very clear what each of these identifiers' purpose is and when they occur anywhere in code, you have a pretty good idea what that variable (or constant) is meant to contain. 

LOCAL_SALES_TAX = .0925 // for Nashville,TN 2023

Comments should be used to explain or clarify code, not in place of writing clear code.

For comparison:

value = dataEntered * 27.3;
static float CONSTANT = 3958.8;
x = java.lang.Math.sqrt( newValue ) / CONSTANT;

This code may be correct, but there is no indication of what are these numbers for or what this code does. If you had to find and fix an error how would you know where to start?

 

As a rule, if there are numbers or string variables anywhere in your code, is it obvious what they are for? Could they be assigned to an identifier to make them clearer and ensure they are defined in only one place?

 

A quick web search found these guidelines for Java coding

  • Use descriptive names for all variables, function names, constants, and other identifiers.
  • Use single letter identifiers only for the counter in loops.
  • Class names start with an upper case letter.
  • Variable names start with a lower case letter. (Variables include parameters, local variables, and data fields. Exception: use UPPER_CASE for constants - final variables.)
  • Method names start with a lower case letter.
  • Multi-word identifiers are internally capitalized.
  • Do not use hyphens or underscores to separate multi-word identifiers (except for constants, which have all upper case letters).
