Getting Started

At a very simple starting point, virtually every computer program:

  • accepts input
  • processes data
  • generates output

input > process > output

Input can be reading a file, enabling keyboard entry or mouse clicks, waiting for a timer to expire, or a pedestrian to press the button to request a WALK signal.

Processing may be simple such adding numbers on a calculator Application or complex such as calculating the required trajectory to launch a satellite into geo-stationary orbit.

Program output may include displaying the contents of a data file on a computer monitor or transmitting a message to an email server or sending a control signal to trigger a measured current to a medical defibrallator.

Not all programs will be sequential, or simple step-by-step, start-to-finish instruction sets, there may be logic loops and conditional decision points, but consider that all systems can be described, at least as a starting point in these simple terms and logic algorithms, or instructions, will make sense.

 

Hello World

There is a simple program that is familiar to all developers. Almost every programming language book, tutorial, or course starts with a simple helloworld program to bridge the human-machine interface. By writing simple programming instructions, we ask the computer to do something.

Computer instructions are written as source code using a programming language. The choice of programming language may vary based on capabilities, system requirements, interoperability, or even programmer or engineer preference. Sometimes the decision is based only on which language the developer has the most familiarity and the most knowledge and comfort.

All of the most popular programming languages in use are either compiled or interpreted. Programs, or computer instruction sets are written using a friendlier, human readable language which is usually similar or familiar to the programmers native spoken language and saved as text files. Text files do not store information about font sizes and styles, just simple letters and numbers and common punctuation which can be written and read from simple editors on just about any computing system. Some languages require that these saved text files, called source code, be run through another program, called a compiler to output another file which contains instructions which are easier, and faster, for the computer to understand.

Other languages run interpreters which read the source code and tell the computer what to do without storing the intermediate computer file.

As an example, a program written in the C programming language, will be stored as a text file, and given a name with a .c file extension, and then run through a C compiler program, such as cc or gcc, and might be output as a binary file which might have a .exe file extension for use on a Windows PC and can run on its own.

In contrast, a program written in the Python language requires the Python interpreter to be running on the host machine when the program runs. In both of these cases (C and Python) the language is widely supported and the source code can be compiled or interpreted and run on Windows, MacOS or Linux computer systems with little to no changes.

Exercise 1: Using the command line

dotnet

Let's walk through the structure and process of writing and executing a C# application.

For this first exercise, use the command line, not the Visual Studio IDE. From Windows, you can use either the traditional DOS command line (CMD) or the more modern and capable Powershell.

C# code is compiled to intermediate language (CIL) and processed (again) by the .NET runtime CLR and optimized for the particular system according to its capabilities. Simiilar to Java programs, these .NET programs are contained and executed within a separate runtime virtual machine environment, insulating the operating system from the application.

From the command line, try: dotnet --help
for a brief list of available commands.

Then dotnet --list-sdks
to view the available installed versions on your system.

This will confirm whether the SDK is installed or not. As of this date, Summer, 2023, .NET 6.0 (LTS), and 7.0 are supported and available for download. The .NET preview release (8.0) is also available for early adopters, but may not be fully tested. Since .NET 5, Microsoft has announced the new release schedule will be each year in November, so the new release should be getting closer to a stable build.

If none are available, browse to https://dotnet.microsoft.com/en-us/download to download and install the SDKs you need. For compatibility with the textbook (10th edition), use the latest version of .NET 6.0. The designator "LTS" indicates Long Term Support.

In my case, there were newer versions for both .NET6 and .NET7, so I installed both. Downloading and running the installer took only a minute or so.

The numbering on software releases is worth learning more about. Typically the first digit indicates a major release (6 vs 7). The next digit is known as a minor release (x.3 vs x.5), and subsequent digits (x.x.205) indicate a build number, often associated with bug fixes or security patches rather than new features or functionality.

You may also note that the .NET6.0 update removed and replaced the older installation, but the .NET7.0 update was simply added to the list of installed SDKs.

If you have already installed Visual Studio, then chances are some .NET SDK is already installed.

From your lab folder, enter: 

dotnet new console --output HelloWorld

This will create a new project folder named "HelloWorld" containing a project (.csproj) file and a (C#) source file, Program.cs.

Open both of these files to view their contents.

Next, from the lab folder (containing the HelloWorld folder), run dotnet build HelloWorld.

This should complete with 0 warnings and 0 errors.

Run dotnet run --project HelloWorld

The code executes and says "Hello" to the console.

Next edit the Program.cs file using a text editor (not a word processor!) like Notepad.

Enter the following code. Upper/Lower case matters!: 

using System;

namespace hello
{
    class Program
    {
        static void Main(string[] args)
        {
            Console.WriteLine("Hello C# Programming World!");
        }
    }
}

  • The single using statement indicates that we will need the System library for the Console object methods.
  • The namespace will contain the class and other elements within the scope of this application.
  • "Program" is the name of the class and matches the name of the file.
  • There is a single static void method in the class called Main(). This is the entry point for the application.
  • string[] args are the parameters that can be passed to the application, in this case from the command line.
  • Finally, the single command, Console.WriteLine() outputs our message to the display. Change it from Hello, World to anything else.

Navigate back to the lab01 folder, build and run it as before.

Starting in C# 9 (released with .NET 6.0), you don't have to explicitly include a Main method in a console application project. Instead, you can use the top-level statements feature to minimize the code you have to write. In this case, the compiler generates a class and Main method entry point for the application.

Read more: https://learn.microsoft.com/en-us/dotnet/csharp/fundamentals/program-structure/top-level-statements

It's important to note that the top-level Main() and using and class and namespace statements are not gone or optional, but in simple cases, if they are left out, the compiler will generate them by default.

Most of the code that we write will include all of these things, that is why it is essential to know what they are, what they mean, and where they go. We'll talk about them more later.
