Relational vs. Non-Relational Databases

By Tamara Pattinson

What are relational databases?

A relational database stores data in tables and rows also referred to as records. The term “relational database” was first used in 1970 by E.F. Codd at IBM in his research paper “A Relational Model of Data for Large Shared Data Banks.”

A relational database works by linking information from multiple tables through the use of “keys.” A key is a unique identifier which can be assigned to a row of data contained within a table. This unique identifier, called a “primary key,” can then be included in a record located in another table when that record has a relationship to the primary record in the main table. When this unique primary key is added to a record in another table, it is called a “foreign key” in the associated table. The connection between the primary and foreign key then creates the “relationship” between records contained across multiple tables.

What are non-relational databases?

In contrast, the non-relational database, or NoSQL database, stores data. However, unlike the relational database, there are no tables, rows, primary keys or foreign keys. Instead, the non-relational database uses a storage model optimized for specific requirements of the type of data being stored.

The five types of non-relational databases are: document data store, column-oriented database, key-value store, document store, and graph database.  Often combinations of these types are used for a single application.

Document data stores
A document data store manages a set of named string fields and object data values in an entity referred to as a “document” typically stored in the form of JSON documents.
The document stores do not require all documents to maintain identical data structures, which provides a great deal of flexibility. It’s easy to see then how this flexibility can be leveraged as an organization’s requirements change.
Columnar (or column oriented) data stores
A columnar data store organizes data into columns, which is conceptually similar to the relational database. The true advantage of a column-family database is in its denormalized approach to structuring sparse data, which comes from its column-oriented approach to storing data.
Key value stores
This is the least complicated of the NoSQL databases. The key-value store is simply a collection of key-value pairs contained within an object.
Document stores
Document stores are a bit more complex than key-value stores. The document store is designed to store everyday documents as is, and they (involve) more complicated querying.
Graph databases
Last is the most complex non-relational database type. It’s designed to efficiently store relations between entities. When data is greatly interconnected, such as purchasing and manufacturing systems or referencing catalogs, graph databases are a good solution.

Instead of the Structure Query Language (SQL) used by relational databases, NoSQL database uses Object-relational-mapping (ORM).

A Database?

    The term database can refer to an application, a dataset, or a computer depending on the context.
  • A database may refer to a software application that is used to store and manage large amounts of data. Examples of database applications include Oracle, MySQL, and Microsoft SQL Server.
  • A database can refer to a collection of data that is organized in a specific way to allow for efficient storage, retrieval, and modification. Databases can be used to store many types of data, including text, images, and numerical data.
  • Or, a database may refer to a physical or virtual storage device that is used to store data. This can include hard drives, solid-state drives, and cloud-based storage services.

Cloud providers commonly provide database services as supported Platform-as-a-Service (PaaS). These are sometimes referred to as DBaaS or Database-as-a-Service.

This unfortunate ambiguity may lead to confusion. And to make things worse, all of these are correct!

The client-server nature of the database application, and the tendency for these installations to be commonly hosted on dedicated computers, makes this service an ideal candidate for cloud providers. And it is no surprise that the leading cloud providers do provide ready-to-implement solution platforms to host datasets as relational or non-relational configurations.

These cloud database solutions are considered Platform-as-a-Service because they are not end-user applications (SaaS), and while they are highly configurable, they do not require software installation and server maintenance as an Infrastructure-as-a-Service (IaaS) would. They are generally considered turn-key utilities, ready for datasets to be written to and utilized.
