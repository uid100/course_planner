Separation of Concerns

The concept has been described many ways, using many terms, but the practice of Separation of Concerns is a principal concern in software engineering. As computing power became more capable and data storage more practical, the size and complexity of computer applications grew. I've personally worked on many legacy code projects that were described as spaghetti code or Jenga code. These nicknames are sometimes used to describe monolithic applications which grow and grow and are intertwined and have so many inherent dependencies, they become very fragile.

Imagine stirring a giant pot of soup and someone adds some salt and pepper, and someone else visits and adds a little garlic, and another person steps in and decides it needs a little cumin, and another contributor shakes in some paprika. As these become mixed in, trying to remove or replace any single ingredient becomes impossible, and if you wanted to remove any one thing, the best course of action is to get a new pot and start over.

This gave way to the concept of multi-layered or n-tiered architectures and frameworks. Instead of the soup pot, we can use a salad bar approach. Take the components that you want, combine them together, each component may interact with one another and even complement each other, but they remain identifiable as individual components.

I may have an application which accesses the card catalog from the local library, makes requests for book availability, perhaps for transfer to another nearby library request, and another application may use that same card catalog for a different application which tracks historical patterns and other metrics so that a single neighborhood library does not end up with 27 copies of a book that has very little interest, and another has 1 copy of a text that is in high regional demand. The idea is that card catalog (or database) can be loosely tied to each of those applications, but still be useful on its own. And if the database is migrated to a new database with the same interface, those applications do not need to be rewritten because of a simple configuration change.

The Model-View-Controller framework is a good example of multi-layered, or n-tiered solutions. If I keep my components small and simple, each doing one thing, then mixing and matching them should be relatively simple. The data model may be reusable by other applications. Many different view representations can reuse the same sets of data. Think about weather. I can display a map view of Northern Florida, or a table of local cities and regional temperatures and rainfall totals, or I can display radar and satellite tracking. All of those are different views, or ways of looking at the same data. These are also called representational states. 

Also, in the same way that different applications can share data sources, applications can use multiple data sources. In other words, in the first case, multiple stores can sell the same products, but a store location might host multiple seasonal, or pop-up vendors.

In object-oriented programming, you'll often hear the expression loosely-coupled, and highly cohesive. If this is new to you, read it again, write it down, commit it to memory. These are more than buzz-words, but they are important discussion topics in any technical interview. It means that my components should be designed and built so that they play well with others, but not dependent on them. My component does exactly what it needs to do, but nothing 'extra.'  The Swiss Army Knife is a great example which of a tool that does 14 things, but none of them very well.

Data-driven Applications

All of this comes together this week with a focus on data models. Think about different representations of the same data. The data itself needs to be well-structured to support this. Strongly-typed views only work if the data models (or data-view models) are correctly built.
