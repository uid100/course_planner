Hash functions

What is a hash function

A hash function is a mathematical function that takes an input (or "message") and returns a fixed-size string of bytes. The output, commonly called the hash code or hash value, is typically a digest that is unique to each unique input.

Hash functions are widely used for various purposes, especially in cryptography.

Hash functions are designed:

  • ... to be Deterministic. For the same input, the hash function should always produce the same output.
  • ... to be Efficient. It should be computationally efficient to compute the hash value for any given input.
  • ... to have a Fixed Output Size. The hash function should produce a fixed-size output, regardless of the size of the input.
  • ... to be Collision Resistant. It should be computationally infeasible to find two different inputs that produce the same hash value. In practice, this property is essential to ensure the integrity of data and the security of hash-based algorithms.
  • ... to have an Avalanche Effect. A small change in the input should result in a significantly different hash value. This property ensures that similar inputs have very different hash codes.

This is a seemingly simple description of what can be very complex mathematical applications.

For Example Passwords

Here's a common use-case. A user sets an access password. Imagine that password can contain upper and lower case values of the 26 letters of the english language plus 10 numeric digits and maybe a hanful of allowable printable characters. To make it more complex, it may be between 10 and 20 characters long. This unknown length combination of perhaps 70 or more pseudo-random characters presents as... (skipping the math...) a whole lot(!) of possible combinations.

If you run this password string through one of our complex hash functions, it produces a presumably unique string that is equally random. Because of the characteristics of the hash function described above, even storing this value as plain text, makes guessing or even calculating the source string very unlikely, or at the least, very computationally complicated.

So when the authorized user returns and re-enters the pre-selected password, and it is again validated against the same hash function, only the exact password will result in a match with the stored password hash.

Common Uses

Hash functions are used in various applications, such as:

  • Data Integrity Verifying the integrity of data by comparing hash values before and after transmission or storage. If the data has been altered even slightly, the hash value will not match.
  • Hash Tables Efficiently organizing and retrieving data in hash tables and other data structures.
  • Digital Signatures Ensuring the authenticity and integrity of messages and documents. Similar to the way that a data transfer can be checked as described above, the source for a message or document can also be checked using hashes.
  • Cryptography Password storage, and other security applications.

 

Common hash functions used in cryptography include

  • MD5 (Message Digest 5)
  • SHA-1 (Secure Hash Algorithm 1)
  • SHA-256
  • SHA-3

Older hash functions like MD5 and SHA-1 are still used in some contexts, they are considered insecure for cryptographic purposes due to vulnerabilities, and more secure hash functions are recommended for new applications.

 

Hash Tables

Hash functions do not need to be complex and unbreakable to be useful or effective. The application will determine the complexity of the hash function.

What is a hash table

A hash table (also known as a hash map) is a data structure that implements an associative array abstract data type (ADT), a structure that can map keys to values. It uses a hash function to compute an index into an array of buckets or slots, from which the desired value can be found.

The idea behind a hash table is to transform the key into a hash code, which is used as an index to access the corresponding value in the array.

Ideally, each unique key should produce a unique hash code, but in practice, collisions (two different keys producing the same hash code) can occur. Various techniques are used to handle collisions.

Two common approaches are

  • Separate chaining, and
  • Open addressing

 

How a hash table works

  1. A hash function takes a key as input and produces a hash code, integer to be used as an index to where the corresponding value should be stored or looked up.
  2. Buckets The hash table typically consists of an array (or vector) of buckets or slots. Each bucket can hold multiple key-value pairs in the case of collisions.
  3. Collision Resolution (when more than one key-value resolves to the same index)
    • Separate Chaining Each bucket contains a linked list (or another data structure) to store multiple key-value pairs with the same hash code. (To hold multiple values at the same index)
    • Open Addressing If a collision occurs, search for the next available slot (the next "open address) in the array (linear probing is one common method) until an empty slot is found. Although the hash function may not find the key in the table immediately, if the search starts there, the algorithm should not have to search far starting from there to find the target.
  4. Table Operations
    • Insert The key-value pair is inserted into the bucket determined by the hash code.
    • Retrieve The hash code is used to find the appropriate bucket, and then the key is searched within the bucket to retrieve the associated value.
    • Delete Similar to retrieval, the key is located in the corresponding bucket, and the key-value pair is removed.

Hash tables provide fast average-case time complexity for basic operations (insertion, retrieval, and deletion), making them efficient for tasks like indexing and searching.

The efficiency depends on the quality of the hash function and the collision resolution strategy employed. Simpler hash functions are more likely for collisions to occur.

Common programming languages often provide built-in implementations of hash tables, such as Python's dict or C++'s std::unordered_map.
