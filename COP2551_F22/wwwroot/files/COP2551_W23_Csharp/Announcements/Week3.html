Code Review

These are what we would commonly call findings in a formal code review. A finding may be something that is incorrect, or doesn't match expected standad coding practices or just opportunities to improve the code.

After the initial code review (or peer review) is complete, the findings are collected and recorded, usually tracked by the Quality Assurance (QA) team. The findings are presented to the creator and all are addressed by either a change to the artifact (the code) or by some other acknowledgement or clarifying response. Not all findings require corrections, but all findings need to be addressed.

By the way, the QA team may track metrics such as how many findings per line of code, how long does it take for findings to be addressed (closed) and maybe how many findings are dismissed or corrected.

After the creator and the reviewer agree that all of the findings have been addressed, the code can be integrated with the test build and the test team may take over.

Of course this process will vary from team to team and project to project. But the general workflow will be similar.

So let's consider my list of findings and you can look at your code and decide if any of these apply. And no, you won't need to make any corrections and resubmit them. After you've been through the list, I'll consider these closed and next time I may take points.

 

Identifiers

Our textbook talks a little about identifiers. In fact every programming textbook will talk about identifiers. I wish it said more.

Rules

The book lists rules for legal C# identifiers. In fact, these rules generally apply to all C-style programming languages, including C/C++/C# and Java. Most of these are common to other languages also, but for now we'll just consider these. According to the book:

  • Must start with a letter [a-z] or [A-Z] or an underscore (_).
  • May also contain digits [0-9]. Can't start with any.
  • May not contain spaces.

But wait, there's more!

These are rules. There are also conventions and best practices.

Conventions

As long as the rules are met, the conventions are really just suggestions. But consider them strong suggestions. Just because your code runs doesn't mean it is correct.

Conventions make your code more consistent, easier to read and easier to understand, and by extension, easier to maintain and less error-prone.

C-style languages, including C# use camelCase formatting for compound names. PascalCase is a specialized form of camelCase in which the first character is capitalized. It is named for the French mathemetician, Blaise Pascal, not for the programming language which also is named for him. Since it is his last name, the capital letter makes sense and is easier to remember.

C# conventions usually expects camelCase for variables, and PascalCase for a class, record, struct, and public type members such as fields, properties, and methods.

There's more, but just remember public -- capitalize, private -- lower case.

Unnamed Constants

Finding #2
Avoid using unnamed constants in your code.

I already discussed this one last week (go back and read it again.) 

valueA = double.Parse(classATextBox.Text) * 15

Why 15? Why is this in the code and not defined as a constant near the top of the code? And if the same constant is used more than once in the code, that is much worse. Please be careful with these. They may be correct or not. They won't cause runtime exceptions, and they can lead to errors that can be very difficult to find and very difficult to correct.

Consistent Types

Finding #3
Variable (and constant) type assignments should match.

Sometimes named constants were used to hold the correct assignments for the ticket prices, but the wrong types were assigned.

This initializes the value 

double CLASS_A_PRICE = 15
but uses an integer, which the compiler, correctly casts to an integer, but implicit casts are not ideal. Try using 
= 15.0
instead to ensure the value is set to a double, or use 
15M
if you are using a Decimal variable type.

Again, not an error, but matching the correct variable types makes your code clearer, and more reliable.

Formatted Output

Finding #5
Check the format of your output.

Two observations here, and again, most of these were really good, these are just some notes to check.

Monetary (price, costs, sales totals) should be formatted as currency. This can be a little complex, but at the least, add a dollar sign in your output.

One more related to output. This one I did not see as a problem for this example, but it becomes a common problem in later exercises, so I'll mention it now. Output results should be written as read-only text boxes, not labels.

The reason is that textboxes, even read-only, can be selected and copy/pasted to other applications. Labels cannot. Try it. If your program has collected data and performed some calculations, you might want to reuse that data and manually typing it from one place to another doesn't make sense.

Tab Order

Finding #6
Set the tab order for your form elements.

This one is a little more advanced and I wouldn't expect these to be correct on the first application, but try this on your next project.

When the application starts is the cursor on the first input box? Can you use the [Tab] key to advance to the next logical input, or control?

More advanced, does the [Enter] key perform the correct default Submit action? Does [Esc] exit the program? These are a little tricky to find, but once you do, it will make your programs better.
