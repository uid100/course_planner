Data Design

You have seen that design documents are are useful for communication and negotiation among stakeholders and for elaboration and clarification of a product. As the design evolves, the solution is formed and having a shared understanding of objectives and approaches is essential to success.

In this module we will develop the conceptual data design from the perspective of both the database and software implementation.

    Consider:
  • What is a conceptual database design
  • What is an entity
  • What is an attribute
  • What is a relationship
  • What is cardinality

While developing your initial concept document, I asked you to think about the data that your application will consume, store, process, and output. Recall from the REST API discussion that there may be multiple visualizations or representations of the data.

ERD (50 points)

Develop a Chen-notation Entity Relationship Diagram (ERD). This is a different notation than the Crowsfoot notation which is also popular. Refer to the reference resources provided in this module.

Your diagram will identify the entities, attributes, relationships, and cardinality relevant to your project.

UML Class Diagram (50 points)

For part 2, develop your initial class diagram to show the implementation of the ERD as software classes.

Each class should include the attributes, the attribute type, visibility, and methods with return type, parameters, and visibility as well as any relationships among other objects.

You can use any drawing tool for this activity. I suggest, and often use the free online draw.io tool. Remember if you choose this, that the tool saves working files as XML format (.drawio) and you'll need to Export your drawings to a common image format (I recommend jpeg) to render them in your GitHub README file.

Submissions

Add both the ERD and Class Diagram(s) to your GitHub repo and README file with corresponding headers in the file. Also update the table of contents in the README so they can be linked quickly in the file.

If the content in your README is appropriately organized with hierarchical headers, adding a table of contents is easy to do. Most Markdown cheatsheets will have instructions.

Enter the link to your GitHub repository when the README is ready for review.
