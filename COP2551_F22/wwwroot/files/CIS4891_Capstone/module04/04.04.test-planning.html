Software Test

Software testing begins at the beginning of a project life cycle and continues throughout. At the start, the test team is concerned with understanding the requirements, and planning for objective ways to test each with the objective of determining whether each has been satisfied.

    There are several types of software testing, including:
  1. Unit Testing: testing individual components or units of the software.
  2. Integration Testing: testing interactions between multiple units or components.
  3. Functional Testing: testing the software against the specified requirements to confirm it functions as intended.
  4. System Testing: testing the entire system as a whole, including all components and interactions.
  5. Acceptance Testing: testing the software to determine whether it is acceptable for release to the end-users.
  6. Regression Testing: testing the software to ensure that changes and new features do not negatively impact existing functionality.
  7. Performance Testing: testing the software to evaluate its speed, stability, and scalability.
  8. Security Testing: testing the software to identify and address security risks and vulnerabilities.
  9. Usability Testing: testing the software to evaluate its user-friendliness and ease of use.
  10. Exploratory Testing: ad-hoc testing with the goal of discovering defects and improving the overall understanding of the software.

Activities

Although testing activities take place throughout the project life cycle, acceptance testing is what is often associated with the so-called testing phase of the SDLC.

Requirements development is subjected to reviews to reduce or eliminate conflicts within them and to ensure they are logical, realistic, and objectively testable.

During solution design, the design is reviewed to ensure the requirements are all being addressed and each requirement may be assigned to a particular component or subsystem of the solution.

Throughout implementation, the developers will consider the solution design and the relevant requirements and implement unit-testing so that each smallest unit, method, module, interface... will satsify the requirements assigned to it.

As the system is integrated, the team will continue to perform unit-testing and test that the system components work together, and individually still operate as designed.

When the solution is built, and all unit testing is passing, and the integration team is satisfied that the system is ready for acceptance testing, the system is tested first to see that each one of the identified and approved requirements is met and passing (validation), then the system should be moved to its intended environment and exercised to see that it satisfies the original intended use.

Test Phases

    The software testing process typically involves the following phases:
  • Planning and Preparation: Define the test objectives, detail strategy and test plan, and preparing the testing environment.
  • Test Analysis and Design: This phase involves defining and documenting the test cases and test data, as well as creating test artifacts such as test scripts, test scenarios, and test data.
  • Test Implementation and Execution: This phase involves executing the test cases and test scenarios and documenting the results. This phase may also involve debugging the software and fixing defects.
  • Result Analysis: This phase involves evaluating the test results, identifying any failures or defects, and determining the cause of the failures.
  • Test Closure: This phase involves documenting the test results, preparing a final report, and archiving the test artifacts. The test closure phase may also involve assessing the effectiveness of the testing process and making recommendations for improvement.

It's worth noting that these phases are iterative and may be repeated multiple times as part of an Agile development process. The goal of the testing process is to identify defects and improve the quality of the software before it is released to end-users.

Verification and Validation (V&V)

Verification and validation are two important concepts in software development and testing.

Verification refers to the process of checking whether the software development process conforms to the standards, regulations, and guidelines. This includes activities such as code reviews, walkthroughs, and inspections, and its goal is to ensure that the software is being developed correctly.

Validation, on the other hand, refers to the process of evaluating software at the end of the development process to determine whether it meets the specified requirements. The goal of validation is to ensure that the software works as intended and satisfies the needs of the end-users. This includes activities such as functional testing, acceptance testing, and user acceptance testing.

Verification asks, "Did we build it right?"

Validation asks, "Did we build the right thing?"

Methods

Some common methods for measurement of success include:

  • inspection
  • demonstration
  • analysis

Inspection is a systematic review of requirements documents.

Demonstration is a step-by-step review with stakeholders and subject matter experts to confirm the requirements are met.

Analysis uses mathematical models or simulations to verify the requirements and provide a comprehensive assessment. For instance if a method outputs a calculated result for given inputs, the test engineers might use another method to calculate the result for comparison.
